# JUnit 5 en Kotlin

## Temario básico de pruebas

JUnit 5 permite crear pruebas automáticas para comprobar que el código funciona como esperas.  
Con él puedes validar resultados, comprobar errores, repetir pruebas y ejecutar el mismo test con varios datos.

Este temario recoge varios ejemplos sencillos para practicar con una clase como `Calculadora`.

---

## 1. Pruebas básicas con `assertEquals`

La forma más simple de empezar es comprobar si un resultado coincide con el valor esperado.  
Para eso se usa:

- `@Test`
- `assertEquals(...)`

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.api.Test

class CalculadoraTest {

    @Test
    fun sumarDosNumeros() {
        val resultado = 2 + 3
        assertEquals(5, resultado)
    }
}
```

### Qué hace

- Marca el método como prueba con `@Test`
- Suma `2 + 3`
- Comprueba que el resultado sea `5`

Si el valor coincide, la prueba pasa.  
Si no coincide, la prueba falla.

---

## 2. Probar varios casos de suma

No basta con probar un único caso.  
Es buena práctica usar varios datos para comprobar comportamientos distintos.

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.api.Test

class CalculadoraTest {

    @Test
    fun sumarPositivos() {
        val resultado = 2 + 3
        assertEquals(5, resultado)
    }

    @Test
    fun sumarConCero() {
        val resultado = 7 + 0
        assertEquals(7, resultado)
    }

    @Test
    fun sumarNegativos() {
        val resultado = -2 + -3
        assertEquals(-5, resultado)
    }
}
```

### Qué se practica aquí

- Caso normal
- Uso del cero
- Números negativos

Esto ayuda a cubrir más situaciones reales.

---

## 3. Probar métodos de una clase

Lo normal es no probar operaciones sueltas, sino funciones reales de una clase.

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.api.Test

class Calculadora {
    fun restar(a: Int, b: Int): Int = a - b
}

class CalculadoraTest {

    @Test
    fun restarDosNumeros() {
        val calc = Calculadora()
        val resultado = calc.restar(8, 3)
        assertEquals(5, resultado)
    }
}
```

### Qué hace

- Crea un objeto de tipo `Calculadora`
- Llama al método `restar`
- Comprueba que `8 - 3` sea `5`

Cada prueba debe centrarse en un comportamiento concreto.

---

## 4. Probar excepciones con `assertThrows`

A veces lo correcto no es obtener un número, sino lanzar un error controlado.  
Para comprobar eso se usa `assertThrows(...)`.

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertThrows
import org.junit.jupiter.api.Test

class Calculadora {
    fun dividir(a: Int, b: Int): Int {
        if (b == 0) throw IllegalArgumentException("No se puede dividir entre cero")
        return a / b
    }
}

class CalculadoraTest {

    @Test
    fun dividirEntreCeroLanzaExcepcion() {
        val calc = Calculadora()

        assertThrows(IllegalArgumentException::class.java) {
            calc.dividir(10, 0)
        }
    }
}
```

### Qué hace

- Intenta dividir entre cero
- Espera una excepción de tipo `IllegalArgumentException`

Si la excepción aparece, la prueba pasa.  
Si no aparece, la prueba falla.

---

## 5. Probar resultados booleanos

Cuando una función devuelve `true` o `false`, se suelen usar:

- `assertTrue(...)`
- `assertFalse(...)`

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertTrue
import org.junit.jupiter.api.Assertions.assertFalse
import org.junit.jupiter.api.Test

class NumeroUtils {
    fun esPar(n: Int): Boolean = n % 2 == 0
}

class NumeroUtilsTest {

    @Test
    fun numeroPar() {
        val utils = NumeroUtils()
        assertTrue(utils.esPar(4))
    }

    @Test
    fun numeroImpar() {
        val utils = NumeroUtils()
        assertFalse(utils.esPar(5))
    }
}
```

### Qué se comprueba

- Que `4` sí es par
- Que `5` no es par

Son pruebas simples pero muy habituales.

---

## 6. Pruebas parametrizadas

Cuando quieres probar lo mismo con varios valores, JUnit 5 permite usar pruebas parametrizadas.  
Así evitas repetir mucho código.

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.params.ParameterizedTest
import org.junit.jupiter.params.provider.CsvSource

class Calculadora {
    fun sumar(a: Int, b: Int): Int = a + b
}

class CalculadoraTest {

    @ParameterizedTest
    @CsvSource(
        "2, 3, 5",
        "10, 0, 10",
        "-1, -1, -2"
    )
    fun sumarVariosCasos(a: Int, b: Int, esperado: Int) {
        val calc = Calculadora()
        assertEquals(esperado, calc.sumar(a, b))
    }
}
```

### Qué hace

- Ejecuta la misma prueba varias veces
- Cambia los datos de entrada en cada ejecución
- Comprueba que el resultado sea el esperado

Es muy útil para probar muchos casos sin copiar varias funciones.

---

## 7. Repetir una prueba varias veces

JUnit 5 también permite repetir un mismo test automáticamente.

### Ejemplo

```kotlin
import org.junit.jupiter.api.Assertions.assertEquals
import org.junit.jupiter.api.RepeatedTest

class CalculadoraTest {

    @RepeatedTest(3)
    fun repetirLaMismaPrueba() {
        val resultado = 2 + 3
        assertEquals(5, resultado)
    }
}
```

### Qué hace

- Ejecuta la misma prueba 3 veces

Esto puede servir para comprobar estabilidad o detectar fallos intermitentes.

---

## 8. Qué conviene practicar primero

Si estás empezando, lo más útil es hacer estas cuatro pruebas:

- Una suma normal
- Una suma con cero
- Una suma con negativos
- Una división entre cero con excepción

Con eso ya practicas lo esencial:

- `@Test`
- `assertEquals(...)`
- `assertThrows(...)`

---

## 9. Resumen

JUnit 5 te permite:

- Comprobar resultados esperados con `assertEquals`
- Verificar valores booleanos con `assertTrue` y `assertFalse`
- Comprobar errores con `assertThrows`
- Repetir pruebas con `@RepeatedTest`
- Ejecutar una misma prueba con varios datos usando `@ParameterizedTest`

La idea general es simple:  
cada test comprueba una pequeña parte del programa para asegurarse de que funciona correctamente.

---

## 10. Estructura recomendada del proyecto

Lo habitual en un proyecto Kotlin con Gradle es separar:

- Código normal en: `src/main/kotlin`
- Pruebas en: `src/test/kotlin`

### Ejemplo

```text
src/
├─ main/
│  └─ kotlin/
│     └─ Calculadora.kt
└─ test/
   └─ kotlin/
      └─ CalculadoraTest.kt
```

Así Gradle entiende qué archivos son código normal y cuáles son pruebas.

## 11. Nota importante

Para que los imports de JUnit funcionen, debes tener añadida la dependencia en `build.gradle.kts`.

### Ejemplo básico

```kotlin
plugins {
    kotlin("jvm") version "2.3.10"
}

repositories {
    mavenCentral()
}

dependencies {
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.3")
}

tasks.test {
    useJUnitPlatform()
}
```

Si no añades esa dependencia, verás errores como:

```text
Unresolved reference: junit
```
